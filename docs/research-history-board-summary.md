# 연구원 이력 관리 앱 정리

> 현재 UI/UX와 제품 의사결정의 최신 기준은 `docs/current-ui-decisions.md`를 우선한다. 이 문서는 초기 구조와 배경 설명용으로 유지한다.

## 1. 앱의 목적

이 앱은 연구원에서 발생하는 중요한 운영 이력을 꾸준히 남기기 위한 내부 기록 도구다.

기록 대상은 문제성 이슈에만 한정하지 않는다.

- 강종/제품 이슈
- 투자/과제 검토
- 설비/시험 이슈
- 고객/품질 이슈
- 중요한 회의 결정 사항
- 제도 개선 논의
- 후속 조치와 검증 결과

목표는 나중에 유사한 상황이 생겼을 때 과거에 어떤 문제가 있었고, 어떤 판단을 했고, 어떻게 대응했는지 빠르게 찾는 것이다.

## 2. 운영 전제

- 민감한 정보가 포함될 수 있으므로 소수의 사용자만 접근한다.
- 입력자는 각 담당자가 아니라 관리자 또는 소수 관리 그룹이다.
- 첫 구현은 로컬 MVP로 시작한다.
- 향후 로그인, 권한, 데이터베이스, 배포 구조를 붙일 수 있게 데이터 구조는 확장 가능하게 둔다.
- 화면은 단순하고 시인성이 좋아야 하며, 무거운 결재 시스템처럼 보이지 않아야 한다.

## 3. 핵심 구조

최종 합의된 데이터 구조는 3단계다.

```text
큰 구분/스티커
  세부 항목
    날짜별 이력
```

예시 1: 기술/품질 이슈

```text
큰 구분/스티커: 표면결함개선
  세부 항목: 압연 조건 변경 후 재발 확인
    2024-05-22 발생 확인
    2024-06-03 대응방안 수립
    2024-06-17 대응 결과 확인
```

예시 2: 회의/제도 결정

```text
큰 구분/스티커: 제도개선논의
  세부 항목: 시험 의뢰 기준 개정 논의
    2024-05-14 회의 결과 정리
    2024-05-28 기준안 초안 작성
    2024-06-10 기준안 확정
```

이 구조의 핵심은 날짜별 기록이 제목 유사도로 연결되는 것이 아니라, 같은 `세부 항목 ID`에 연결된다는 점이다.

## 4. 큰 구분/스티커

큰 구분은 여러 세부 항목을 묶는 운영상의 분류다. 날짜별 이력 화면에서는 작은 스티커로 보인다.

예시:

- 표면결함개선
- 제도개선논의
- 시험조건
- 투자검토
- 고객협의
- 유사사례

스티커는 날짜별 이력 행의 첫 줄에 날짜와 함께 표시한다.

```text
2024-05-22  [표면결함개선]  진행중
압연 조건 변경 후 재발 확인
```

색상은 과도하지 않게 저채도 청록, 회색, 연녹색 계열로 제한한다. 같은 큰 구분은 같은 스티커명과 색을 유지한다.

## 5. 세부 항목

세부 항목은 실제 진행 상태를 갖는 단위다.

세부 항목은 다음 중 하나일 수 있다.

- 문제/이슈
- 회의/결정
- 과제/투자
- 고객/품질
- 기타 운영 기록

세부 항목 예시:

- 압연 조건 변경 후 재발 확인
- 표면 검사 데이터 분석
- 시험 의뢰 기준 개정 논의
- 투자 범위 확정 회의

5월 22일 기록과 6월 3일 기록이 같은 건인지 판단하는 기준은, 둘이 같은 세부 항목에 연결되어 있는지다.

## 6. 날짜별 이력

날짜별 이력은 세부 항목 아래에 쌓이는 실제 기록이다.

날짜별 이력은 다음 내용을 담는다.

- 날짜
- 연결된 세부 항목
- 요약
- 상세 내용
- 상태 변경 여부
- 남은 리스크 또는 후속 조치
- 다음 확인일
- 첨부 파일 또는 참고 링크

날짜별 이력은 기본적으로 사실 기록이다. 상태는 세부 항목이 갖고, 날짜별 이력은 그 상태가 왜 바뀌었는지 근거를 남긴다.

## 7. 상태 판단 방식

상태는 세부 항목 단위로 관리한다.

공통 상태 예시:

- 검토중
- 진행중
- 결정됨
- 후속조치중
- 완료
- 보류

기술/품질 이슈 상태 예시:

```text
발생 -> 원인검토 -> 조치중 -> 검증 -> 완료
```

회의/결정 사항 상태 예시:

```text
논의중 -> 결정됨 -> 후속조치중 -> 완료
```

큰 구분의 상태는 세부 항목들의 상태를 종합해 계산한다.

```text
하나라도 진행 중인 세부 항목이 있으면 큰 구분은 진행중
모든 세부 항목이 완료이면 큰 구분은 완료 후보
관리자가 완료 확인 이력을 남기면 큰 구분은 완료
보류 항목만 남아 있으면 큰 구분은 보류 또는 관찰중
```

큰 구분은 자동으로 바로 완료되지 않는다. 관리자가 최종 확인 이력을 남겨야 완료 처리된다.

## 8. 새 이력 추가 방식

새 이력 추가는 자유 입력부터 시작하지 않는다.

기본 흐름:

```text
대분류/하위 주제 선택
-> 큰 구분/스티커 선택
-> 기존 세부 항목 추천/선택
-> 날짜별 이력 작성
-> 이 기록으로 세부 항목 상태를 바꿀지 선택
```

세부 항목 상세에서 `후속 이력 추가`를 누르면 큰 구분과 세부 항목이 자동 선택된다.

전체 `이력 추가`에서 시작하면 앱이 기존 세부 항목 후보를 먼저 보여준다.

추천 기준:

- 큰 구분
- 하위 주제
- 제목 키워드
- 스티커명
- 태그
- 설비
- 고객
- 상태
- 최근 날짜

애매한 기록은 `검토 필요`로 임시 등록하고, 이후 관리자가 세부 항목 연결을 바꾸거나 병합할 수 있어야 한다.

## 9. 첫 화면

첫 화면은 보고 싶은 종류를 빠르게 선택하는 진입점이다.

대분류 카드:

- 강종/제품
- 투자/과제
- 설비/시험
- 고객/품질

각 대분류 카드 아래에는 하위 주제 카드가 보인다.

예시:

- 강종/제품: HPF, STS, 전기강판, 후판
- 투자/과제: 신규 투자, 국책 과제, 공동 연구, 예산/일정
- 설비/시험: 시험 장비, 분석 의뢰, Pilot, 안전/점검
- 고객/품질: 고객 클레임, 품질 편차, 인증/규격, 양산 적용

각 하위 주제 카드에는 다음 정도만 간단히 보여준다.

- 최근 이력 날짜
- 미완료 항목 수

## 10. 상세 화면

하위 주제를 클릭하면 별도 상세 페이지로 이동한다.

상세 페이지는 두 영역으로 구성한다.

```text
왼쪽: 날짜별 이력 리스트
오른쪽: 선택한 날짜 이력 상세
```

왼쪽 날짜별 이력 리스트:

- 검색창
- 필터 버튼
- 날짜
- 큰 구분 스티커
- 상태
- 이력 제목
- 짧은 요약

오른쪽 선택 이력 상세:

- 큰 구분/세부 항목 제목
- 태그
- 상태 버튼
- 주요 메타 정보
- 같은 세부 항목의 이력 타임라인
- 선택한 날짜의 상세 내용
- 조치 사항
- 남은 리스크
- 첨부 파일
- 관련 항목

오른쪽 내부 이력 목록은 스티커 없이 날짜와 제목만 보이는 단순 타임라인 형태를 유지한다.

## 11. 검색과 필터

날짜별 이력 목록 상단에는 검색창과 필터 버튼을 둔다.

검색 대상:

- 큰 구분 제목
- 스티커명
- 세부 항목 제목
- 날짜별 이력 요약
- 상세 내용
- 태그

필터 후보:

- 상태
- 완료/미완료
- 큰 구분/스티커
- 유형
- 기간
- 설비
- 고객

검색과 필터는 날짜별 이력 시스템을 바꾸지 않고, 표시되는 행만 좁혀 보여준다.

## 12. 화면 기준 목업

현재 기준 목업:

- 전체 흐름: `docs/mockups/issue-board-hybrid-home-subtopic-detail-v1.png`
- 상세 페이지: `docs/mockups/issue-detail-date-ledger-v5-home-button.png`
- 상세 HTML: `docs/mockups/issue-detail-date-ledger-v5-home-button.html`

현재 상세 기준:

- 바깥 왼쪽은 날짜별 이력 리스트다.
- 바깥 왼쪽 행의 첫 줄에는 날짜, 스티커, 상태가 함께 보인다.
- 오른쪽은 선택한 날짜 이력 상세다.
- 오른쪽 내부의 이력 목록은 사진처럼 단순 타임라인이다.
- 오른쪽 내부 타임라인에는 스티커를 표시하지 않는다.

## 13. MVP 범위

첫 구현에 포함한다.

- 대분류/하위 주제 첫 화면
- 하위 주제별 날짜별 이력 리스트
- 큰 구분/스티커
- 세부 항목
- 날짜별 이력
- 기존 세부 항목 추천
- 후속 이력 추가
- 상태 변경 여부 선택
- 큰 구분 상태 자동 계산
- 검색과 기본 필터 UI
- 관련 항목 표시
- 오래 열린 미완료 항목 표시
- 로컬 저장
- JSON 가져오기/내보내기

첫 구현에서 제외한다.

- 조직 전체 인증/권한
- 다중 사용자 실시간 편집
- 이메일/메신저 알림
- 복잡한 결재 라우팅
- 외부 문서관리 시스템 연동

## 14. 중요한 결정 사항

- 앱은 "이슈 관리"보다 "이력 관리"에 가깝다.
- 날짜별 이력 화면을 기본 읽기 방식으로 유지한다.
- 같은 건인지 판단하는 기준은 제목 유사도가 아니라 세부 항목 ID다.
- 큰 구분은 스티커로 표시한다.
- 상태는 세부 항목이 갖고, 큰 구분은 세부 항목 상태를 종합한다.
- 모든 세부 항목이 완료되어도 큰 구분은 바로 완료되지 않고 완료 후보가 된다.
- 최종 완료는 관리자의 완료 확인 이력으로 확정한다.
- 회의 결정 사항도 문제성 이슈와 같은 구조로 기록한다.
