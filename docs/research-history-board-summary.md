# 연구원 이력 관리 앱 요약

> 최신 UI/UX와 제품 의사결정의 기준은 `docs/current-ui-decisions.md`를 우선한다.
> 이 문서는 프로젝트 배경, 현재 정보 구조, 운영 방향을 빠르게 이해하기 위한 요약 문서다.

## 1. 앱의 목적

이 앱은 연구원 내부에서 발생하는 중요한 기술, 운영, 제도, 투자, 시험 관련 이력을 꾸준히 남기기 위한 내부 기록 도구다.

목표는 나중에 유사한 상황이 생겼을 때 다음 질문에 빠르게 답하는 것이다.

- 어떤 이슈가 있었는가
- 어느 범위에서 발생했는가
- 어떤 판단과 조치가 있었는가
- 현재는 접수, 진행, 종료 중 어디에 있는가
- 다음 작업이나 향후 계획은 무엇인가

이 앱은 공개 게시판, 민원 게시판, 결재 시스템이 아니다. 소수 관리자 또는 관리 그룹이 이력을 일관된 형태로 남기고 다시 찾기 쉽게 만드는 것이 핵심이다.

## 2. 현재 정보 구조

현재 기준 정보 구조는 5단계다.

```text
대분류
  하위 주제
    이슈
      세부 항목
        날짜별 이력
```

예시는 다음과 같다.

```text
강종/제품
  STS
    STS-430 표면 결함 재발
      시험 조건 편차 원인 검토
        2026-06-20 기록
```

날짜별 이력은 제목 유사도로 묶지 않는다. 같은 이슈인지 판단하는 기준은 같은 `issueGroupId`, `detailIssueId`에 연결되어 있는지다.

## 3. 주요 용어

`대분류`는 가장 큰 운영 분류다. 현재 MVP 기준은 다음이다.

```text
강종/제품
투자/과제
설비/시험
제도/운영
```

`하위 주제`는 대분류 아래의 업무 영역이다. 예를 들어 `강종/제품` 아래에는 `HPF`, `STS`, `전기강판`, `후판`이 있다.

`이슈`는 여러 날짜별 이력을 묶는 큰 사건 또는 관리 대상이다.

`세부 항목`은 이슈 안에서 실제로 추적하는 하위 작업 또는 판단 단위다.

`날짜별 이력`은 특정 날짜에 남기는 실제 기록이다.

`업무 라벨`은 이슈를 빠르게 식별하기 위한 짧은 분류명이다. 예: `시험조건`, `표면결함개선`. 과거 문서의 `스티커` 표현은 현재 UI에서는 `업무 라벨`로 이해한다.

## 4. 상태와 유형

대표 상태는 반드시 3단계로 단순화한다.

```text
접수 / 진행 / 종료
```

목록, 대시보드, 보고서의 기본 상태 표현은 이 3단계를 사용한다.

세부 단계는 대표 상태 아래에서 기록을 더 자세히 설명하기 위한 보조 값이다.

```text
접수: 발생
진행: 원인검토 / 조치중 / 검증 / 보류
종료: 해결
```

`유형`은 상태가 아니라 기록의 성격이다. 예시는 다음과 같다.

```text
회의 / 시험 / 분석 / 보고 / 조치 / 승인 / 고객대응 / 기타
```

세부 단계와 유형 후보는 MVP에서는 코드 상수로 제공하지만, 운영 단계에서는 관리자 모드에서 추가, 삭제, 이름 변경, 정렬할 수 있어야 한다.

## 5. 날짜별 이력

날짜별 이력은 사실 기록이다. 기본 항목은 다음이다.

- 날짜
- 대표 상태
- 세부 단계
- 유형
- 요약
- 상세 내용
- 향후 계획
- 첨부 URL
- 이슈 대표 상태 반영 여부

`이슈 대표 상태 반영`은 모든 기록에 강제로 적용하지 않는다. 회의록, 참고 시험, 단순 분석 기록처럼 이슈 상태를 바꾸지 않는 기록이 많기 때문에 사용자가 필요할 때만 켠다.

`상세 내용`에는 사용자가 직접 적은 진행 사항이나 조치 내용을 보여준다. 선택한 상태, 세부 단계, 유형 값은 본문에 문장처럼 나열하지 않고 날짜와 제목 옆의 작은 배지로 보여준다.

## 6. 화면 구조

첫 화면은 진입 화면이다. 사용자가 대분류와 하위 주제를 빠르게 선택하는 것이 목적이다.

하위 주제 상세 화면은 다음 구성을 기준으로 한다.

```text
상단: 제목, 이슈 현황, 처리 지연 이슈, 최근 7일 갱신
왼쪽: 날짜별 이력 / 이슈별 모음
오른쪽: 선택된 이력 상세
```

모바일 또는 iPad/Surface Pro 세로처럼 목록과 상세가 위아래로 쌓이는 폭에서는 상세를 기본 접힘 상태로 둔다. 날짜별 이력이나 이슈를 누르면 상세가 펼쳐지고 그 위치로 스크롤된다. 상세를 닫으면 목록 영역으로 자연스럽게 돌아간다.

## 7. 이력 추가

이력 추가 흐름은 다음 순서를 유지한다.

```text
대분류
-> 하위 주제
-> 이슈
-> 세부 항목
-> 기록 내용
```

각 단계에서는 기존 항목 선택과 새 항목 추가가 같은 자리에서 가능해야 한다. 항목이 많아질 수 있으므로 큰 카드 나열형 UI를 기본으로 쓰지 않고, 내려 펼쳐지는 step/select 흐름을 사용한다.

대분류와 하위 주제는 완전히 고정된 값이 아니다. 새 대분류와 새 하위 주제로 기록할 수 있다. 다만 기존 항목의 이름 변경, 순서 변경, 삭제, 숨김은 이력 추가 창이 아니라 관리자 모드의 마스터 데이터 관리에서 다룬다.

## 8. 데이터 관리

MVP는 로컬 앱으로 운영한다. 데이터는 브라우저 `localStorage`에 저장한다.

데이터 관리 기능은 넓은 화면에서만 보여준다. 태블릿보다 작은 화면에서는 이력 조회와 등록 흐름을 우선하고, 데이터 관리 카드는 숨긴다.

내보내기와 가져오기는 두 가지로 구분한다.

- JSON: 전체 백업과 복원용
- Excel `.xlsx`: 운영자가 표 형태로 이력 행을 검토, 수정, 추가하기 위한 보조용

Excel은 CSV가 아니라 실제 `.xlsx` 형식을 사용한다. 현재 구현은 앱에서 생성한 `.xlsx` 구조를 기준으로 가져오기를 처리한다. 외부에서 임의로 만든 복잡한 Excel 파일까지 지원해야 하면 전용 Excel 라이브러리 도입을 별도로 검토한다.

## 9. 보고서 방향

보고서 메뉴는 새 데이터를 만드는 기능이 아니다. 기존 이력을 범위와 템플릿에 맞게 재구성해 보여주는 기능으로 설계한다.

예상 흐름은 다음이다.

- 보고 범위 선택: 대분류, 하위 주제, 이슈, 기간
- 포함 항목 선택: 상세 내용, 향후 계획, 첨부 URL, 상태 변경 이력
- 템플릿 선택: 주간 보고, 이슈 요약, 처리 지연 이슈, 완료 이력
- 출력: 화면 미리보기, Markdown/JSON 내보내기, 추후 문서 파일 내보내기

보고서에서 이슈 상태나 원본 이력을 직접 바꾸지 않는다.
