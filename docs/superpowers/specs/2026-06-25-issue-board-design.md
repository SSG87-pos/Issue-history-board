# 연구원 이슈 보드 설계안

## 목적

연구원에서 발생하는 여러 이슈를 소수의 사용자가 꾸준히 기록하고, 이후 비슷한 문제가 생겼을 때 이전 이슈와 해결 이력을 빠르게 찾아볼 수 있는 앱을 만든다.

이 앱은 공개 게시판이나 무거운 결재 시스템이 아니다. 관리자가 이슈 이력을 정리하고 축적하는 조용한 운영 기록장에 가깝다.

다만 기록 대상은 문제성 이슈에만 한정하지 않는다. 중요한 회의에서 결정된 사항, 제도 개선 논의, 투자/과제 판단, 고객 협의 결과처럼 나중에 다시 찾아봐야 하는 운영 기록도 같은 방식으로 남긴다. 따라서 제품 개념은 좁은 의미의 "이슈 관리"보다 "연구원 이력 관리"에 가깝다.

첫 MVP가 답해야 하는 질문은 다음과 같다.

- 최근 어떤 이슈가 있었는가?
- 아직 해결되지 않은 이슈는 무엇인가?
- 오래 열린 미해결 이슈는 무엇인가?
- 특정 이슈는 시간 순서대로 어떻게 진행되었는가?
- 새 기록을 추가할 때 기존 이슈의 후속 이력인가, 완전히 새로운 이슈인가?
- 함께 봐야 할 연관 이슈는 무엇인가?
- 중요한 회의 결정 사항과 후속 조치는 어디까지 진행되었는가?

## 사용자와 운영 방식

이 앱은 민감한 내부 정보를 담을 수 있으므로 소수의 제한된 사용자만 본다.

첫 MVP의 주 입력자는 각 이슈 담당자가 아니라 관리자 또는 소수의 관리 그룹이다. 관리자가 여러 출처의 이슈를 통합해 일관된 방식으로 기록한다.

첫 구현은 로컬 MVP로 시작할 수 있다. 다만 나중에 로그인, 권한, 데이터베이스, 배포 구조를 붙일 수 있도록 데이터 구조는 확장 가능하게 잡는다.

## 핵심 방향

합의된 방향은 다음과 같다.

- 통합 이력 레지스터
- 원인과 조치 중심의 이력 흐름
- 팀 운영 보드
- 깔끔하고 단순한 화면
- 첫 화면은 대분류와 하위 주제 중심
- 하위 주제 상세의 기본 읽기 방식은 날짜별 이력 목록
- 새 이력 추가 시 기존 큰 구분과 세부 항목 선택을 우선
- 자동 추천과 수동 조정을 함께 쓰는 이력 묶음

## 첫 화면

첫 화면은 보고 싶은 이슈 종류를 쉽게 선택할 수 있어야 한다. 복잡한 지표판이 아니라 빠르게 접근하는 시작점이어야 한다.

첫 화면에는 대분류 카드가 있고, 각 대분류 카드 안에 하위 주제 카드가 바로 보인다.

초기 대분류 예시는 다음과 같다.

- 강종/제품
- 투자/과제
- 설비/시험
- 고객/품질

하위 주제 예시는 다음과 같다.

- 강종/제품: HPF, STS, 전기강판, 후판
- 투자/과제: 신규 투자, 국책 과제, 공동 연구, 예산/일정
- 설비/시험: 시험 장비, 분석 의뢰, Pilot, 안전/점검
- 고객/품질: 고객 클레임, 품질 편차, 인증/규격, 양산 적용

하위 주제는 고정 코드가 아니라 관리자가 추가, 수정, 숨김, 정렬할 수 있는 마스터 데이터로 둔다.

각 하위 주제 카드는 정보가 많지 않아야 한다. 다음 두 가지만 기본으로 보여준다.

- 가장 최근 이슈 발생일 또는 최근 업데이트일
- 미해결 이슈 묶음 수

첫 화면 하단 또는 측면에는 다음을 가볍게 보여준다.

- 오래 열린 미해결 이슈
- 대분류별 또는 상태별 미해결 추이 그래프

## 대분류와 하위 주제 흐름

사용자가 하위 주제를 클릭하면 해당 하위 주제의 날짜별 이력 목록으로 이동한다.

예시 경로:

```text
강종/제품 -> STS
```

이 화면에서는 STS 관련 날짜별 이력이 목록으로 보인다. 각 행은 날짜 기준으로 정렬되지만, 행 제목 안에 이슈 묶음명이 함께 드러난다. 특정 이력 행을 클릭하면 오른쪽 상세에서 그 날짜의 기록을 크게 보고, 같은 이슈 묶음에 속한 이전/이후 기록을 함께 확인한다.

날짜별 이력 목록 상단에는 검색창과 필터 버튼을 둔다. 검색은 날짜별 이력 시스템을 바꾸지 않고, 표시되는 이력 행만 좁히는 기능이다. 검색 대상은 이슈 묶음 제목, 스티커명, 이력 요약, 상세 내용, 태그를 포함한다. 필터는 상태, 미해결 여부, 스티커/묶음, 기간을 기준으로 확장할 수 있다.

각 이력 행은 다음 정보를 보여준다.

- 날짜
- 이슈 묶음 제목 또는 구분명 스티커
- 해당 날짜의 짧은 요약
- 단계 또는 현재 상태
- 미해결 표시 또는 다음 확인일

같은 이슈로 묶이는 기록은 `표면결함개선`, `제도개선논의`, `유사사례`처럼 짧은 스티커 형태로 표시한다. 스티커는 행 제목을 길게 만들지 않기 위한 보조 시각 요소이며, 날짜별 이력 행의 첫 줄에 날짜와 함께 배치한다. 색상은 과도한 원색이 아니라 저채도 청록, 회색, 연녹색 계열로 제한한다. 같은 묶음은 같은 색상과 같은 스티커명을 유지한다.

예시:

```text
2024-05-22 | STS-430 표면 결함 재발) 압연 조건 변경 후 재발 확인 | 진행중
2024-05-17 | STS-430 표면 결함 재발) 원인 가설 검증 결과 | 원인검토
2024-05-14 | 제도 개선 논의) 시험 의뢰 기준 회의 결과 정리 | 검토중
```

## 큰 구분, 세부 항목, 날짜별 이력

데이터 구조는 큰 구분, 세부 항목, 날짜별 이력을 분리한다. 이 3단계를 분리해야 앱이 "5월 22일 기록과 6월 3일 기록이 같은 건인지"를 제목 유사도로 추측하지 않고 안정적으로 연결할 수 있다.

큰 구분은 여러 세부 항목을 묶는 운영상의 분류다. 날짜별 이력 화면에서는 스티커로 보인다.

예시:

```text
표면결함개선
제도개선논의
유사사례
```

세부 항목은 실제로 상태를 갖고 진행 여부를 판단하는 단위다. 문제성 이슈일 수도 있고, 회의에서 결정된 후속 조치, 제도 개선 과제, 투자 검토 항목일 수도 있다.

예시:

```text
압연 조건 변경 후 재발 확인
표면 검사 데이터 분석
시험 의뢰 기준 개정 논의
투자 범위 확정 회의
```

날짜별 이력은 해당 세부 항목 아래에 쌓이는 기록이다.

예시:

```text
2024-05-22 발생 확인
2024-06-03 대응방안 수립
2024-06-17 대응 결과 확인
```

데이터는 세부 이슈 ID를 기준으로 연결되지만, 사용자가 보는 기본 목록은 날짜별 이력 중심이다. 이렇게 하면 매일 어떤 일이 있었는지 자연스럽게 읽히면서도 같은 사건의 기록이 흩어지지 않는다.

```text
큰 구분/스티커: 표면결함개선
  세부 항목: 압연 조건 변경 후 재발 확인
    날짜별 이력: 2024-05-22 발생 확인
    날짜별 이력: 2024-06-03 대응방안 수립
    날짜별 이력: 2024-06-17 대응 결과 확인

큰 구분/스티커: 제도개선논의
  세부 항목: 시험 의뢰 기준 개정 논의
    날짜별 이력: 2024-05-14 회의 결과 정리
    날짜별 이력: 2024-05-28 기준안 초안 작성
    날짜별 이력: 2024-06-10 기준안 확정
```

## 상태 판단 방식

상태는 날짜별 이력이 아니라 세부 이슈가 가진다. 날짜별 이력은 그 상태가 바뀐 이유와 근거를 남기는 기록이다.

세부 이슈 상태 예시:

- 발생
- 원인검토
- 조치중
- 검증
- 완료
- 보류

큰 구분의 상태는 그 아래 세부 이슈들의 상태를 종합해서 계산한다.

```text
세부 이슈 중 하나라도 발생/원인검토/조치중/검증이면 큰 구분은 진행중
모든 세부 이슈가 완료이면 큰 구분은 해결 후보
관리자가 최종 확인 이력을 작성하면 큰 구분은 해결
보류 세부 이슈만 남아 있으면 큰 구분은 보류 또는 관찰중
```

큰 구분이 자동으로 `해결 후보`가 되더라도 바로 `해결`로 닫지 않는다. 연구/품질 이슈는 조치가 끝났어도 재발 없음 확인, 고객/현장 확인, 추가 모니터링 종료 같은 최종 판단이 필요할 수 있으므로, 관리자가 `해결 확인 이력`을 남겨야 큰 구분이 해결된다.

## 같은 세부 이슈 연결 방식

5월 22일 기록과 6월 3일 기록이 같은 세부 이슈인지 판단하는 기준은 제목 유사도가 아니라 같은 세부 이슈 ID에 연결되어 있는지 여부다.

앱의 연결 방식은 다음을 따른다.

- 세부 이슈 상세에서 `후속 이력 추가`를 누르면 새 날짜별 이력은 자동으로 같은 세부 이슈에 연결된다.
- 전체 `이력 추가`에서 시작하면 큰 구분을 먼저 고르고, 그 아래 기존 세부 이슈 후보를 먼저 보여준다.
- 후보 추천 기준은 큰 구분, 하위 주제, 제목 키워드, 스티커, 태그, 설비, 고객, 상태, 최근 날짜다.
- 관리자가 기존 세부 이슈를 선택하면 새 날짜별 이력이 그 세부 이슈 아래에 붙는다.
- 맞는 세부 이슈가 없을 때만 새 세부 이슈를 만든다.
- 애매한 기록은 `검토 필요` 상태로 임시 등록하고, 나중에 세부 이슈 연결을 바꾸거나 병합할 수 있어야 한다.

## 새 이력 추가 흐름

새 기록을 추가할 때 제목을 바로 자유 입력하게 만들면 안 된다. 그렇게 하면 같은 사건이 다음처럼 흩어질 수 있다.

- STS-430 표면 결함
- STS430 표면 문제
- STS-430 표면 결함 재발

따라서 새 이력 추가는 선택된 대분류와 하위 주제에서 시작한다.

예시 흐름:

1. 사용자가 `강종/제품 -> STS`를 선택한다.
2. `이력 추가`를 누른다.
3. 앱이 큰 구분/스티커를 먼저 선택하게 한다.
4. 선택한 큰 구분 아래의 기존 세부 이슈를 검색하거나 추천한다.
5. 추천 기준은 하위 주제, 키워드, 태그, 설비, 고객, 상태, 최근 업데이트일이다.
6. 맞는 세부 이슈가 있으면 그 아래에 새 날짜별 이력을 추가한다.
7. 맞는 세부 이슈가 없을 때만 새 세부 이슈를 만든다.
8. 이 날짜별 이력이 세부 이슈 상태를 바꾸는 기록인지 선택한다.

이 흐름을 기본으로 해야 같은 이름의 이력이 안정적으로 이어진다.

세부 이슈 상세에서 `후속 이력 추가`를 누르는 경우에는 큰 구분과 세부 이슈가 자동으로 선택된 상태로 이력 입력이 열린다.

## 하위 주제 상세 화면

하위 주제 상세 화면은 `강종/제품 > STS`처럼 특정 하위 주제를 선택했을 때 열리는 별도 페이지다. 이 페이지의 왼쪽 영역에는 해당 하위 주제의 날짜별 이력 리스트가 나온다. 이 리스트는 이슈 묶음별 메뉴가 아니라, 운영자가 실제로 쌓아온 기록을 날짜순으로 길게 읽는 영역이다.

예시 날짜별 이력 리스트:

```text
2024-05-22 | STS-430 표면 결함 재발) 압연 조건 변경 후 재발 확인
2024-05-17 | STS-430 표면 결함 재발) 원인 가설 검증 결과
2024-05-14 | 제도 개선 논의) 시험 의뢰 기준 회의 결과 정리
2024-05-10 | STS-430 표면 결함 재발) 표면 검사 데이터 분석
```

왼쪽 이력 리스트에서 특정 날짜 기록을 클릭하면 오른쪽 영역에 그 날짜 이력의 상세가 열린다. 오른쪽 영역은 사용자가 제시한 두 번째 참고 이미지처럼 이슈 제목, 태그, 상태, 메타 정보, 선택 이력 본문, 첨부 파일, 관련 이슈를 하나의 문서형 프레임으로 보여준다.

오른쪽 영역 안에는 같은 이슈 묶음에 속한 이전/이후 이력도 함께 보인다. 이 목록은 첫 번째 참고 이미지처럼 세로 타임라인 형태를 유지하며, 내부 타임라인은 날짜와 이력 제목만 보여주는 단순한 형태로 둔다. 예를 들어 왼쪽 날짜별 이력에서 `2024-05-22 | 표면결함개선 | 압연 조건 변경 후 재발 확인`을 클릭하면 오른쪽에서 5월 22일 상세를 크게 보고, 그 안의 타임라인에서 다음 흐름을 함께 확인한다.

```text
2024-04-28 최초 보고 및 현상 확인
2024-05-10 표면 검사 데이터 분석
2024-05-17 원인 가설 검증 결과
2024-05-22 압연 조건 변경 후 재발 확인
```

즉 화면 전체 기준으로는 `왼쪽 날짜별 이력 리스트`와 `오른쪽 선택 이력 상세`의 두 영역이다. 이슈 묶음은 별도 왼쪽 메뉴가 아니라 오른쪽 상세 안에서 "같은 이슈의 이력"과 "관련 이슈"를 보여주기 위한 연결 기준으로 사용한다.

## 묶음 타임라인

상세 화면은 날짜순 흐름을 유지하면서도 같은 주제의 이력을 블록으로 묶을 수 있어야 한다.

예시 묶음:

- 원인검토
- 조치 및 검증
- 고객 협의
- 투자 판단
- 후속 확인

묶음은 자동 추천과 수동 조정을 함께 사용한다.

자동 추천 기준:

- 단계 또는 상태
- 태그
- 하위 주제
- 유사 키워드
- 기존 큰 구분 제목
- 기존 세부 이슈 제목

수동 조정이 반드시 필요하다. 민감하거나 미묘한 내부 이슈는 관리자의 판단이 중요하기 때문이다. 관리자는 묶음 이름을 바꾸거나, 특정 이력 항목을 다른 묶음으로 옮기거나, 묶지 않은 독립 항목으로 둘 수 있어야 한다.

## 연관 이슈

상세 화면에는 선택한 이슈와 함께 볼 만한 연관 이슈를 보여준다.

연관 이슈 추천 기준:

- 같은 대분류
- 같은 하위 주제
- 공통 태그
- 같은 강종, 설비, 고객, 원인 키워드
- 유사한 제목 또는 요약

연관 이슈는 참고로 보여줄 뿐 자동 병합하지 않는다. 두 기록이 정말 같은 이슈인지 판단하고 묶는 권한은 관리자에게 있어야 한다.

## 초기 필드

큰 구분/스티커 필드:

- 제목
- 대분류
- 하위 주제
- 스티커명
- 스티커 색상 톤
- 상태 자동 계산값
- 관리자가 확정한 최종 상태
- 해결 확인일
- 해결 확인 이력 ID
- 민감 정보 표시
- 보관 여부

세부 이슈 필드:

- 제목
- 큰 구분 ID
- 상태
- 최초 발생일
- 최근 업데이트일
- 현재 한 줄 요약
- 태그
- 관련 설비
- 관련 고객
- 우선순위
- 담당/작성자
- 완료일
- 완료 판단 메모
- 민감 정보 표시
- 보관 여부

날짜별 이력 필드:

- 날짜
- 연결된 세부 이슈 ID
- 단계 또는 상태 변경값
- 상태 변경 여부
- 요약
- 상세 내용
- 남은 리스크
- 다음 확인일
- 묶음 블록 이름
- 첨부 또는 참고 링크
- 생성/수정 정보

MVP의 입력 양식은 짧아야 한다. 대부분의 일상 기록은 날짜별 이력 추가 양식에서 끝나야 한다. 다만 날짜별 이력 추가 양식에는 `연결할 세부 이슈`와 `이 기록으로 세부 이슈 상태를 변경할지`를 반드시 선택할 수 있어야 한다.

## 상태 흐름

기본 상태 흐름은 원인과 조치 중심으로 둔다.

- 발생
- 원인검토
- 조치중
- 검증
- 해결
- 보류

투자나 과제 이슈는 나중에 의사결정 관련 필드를 추가할 수 있다. 첫 MVP에서는 무거운 승인 흐름을 만들지 않는다.

## 민감 정보와 접근

첫 MVP는 소수의 신뢰된 사용자만 보는 것을 전제로 한다. 로컬 MVP에서 완전한 인증을 구현하지 않더라도, 향후 권한과 접근 제어를 붙일 수 있게 설계한다.

기본 원칙:

- 공개 배포하지 않는다.
- 핵심 기록은 외부 네트워크 의존 없이 볼 수 있게 한다.
- 이슈 묶음에 민감 정보 표시를 둘 수 있게 한다.
- 내보내기는 자동이 아니라 사용자가 의도적으로 실행하게 한다.

## MVP 범위

첫 구현에 포함할 항목:

- 대분류와 하위 주제 중심의 첫 화면
- 대분류/하위 주제 마스터 데이터
- 하위 주제별 날짜순 이력 목록
- 이슈 묶음 생성
- 새 이력 추가 전 기존 큰 구분과 세부 이슈 검색 및 추천
- 날짜별 이력 생성
- 전체 이슈 타임라인이 보이는 상세 화면
- 묶음 타임라인 블록
- 연관 이슈 추천
- 오래 열린 미해결 이슈 영역
- 간단한 대분류/상태 그래프
- 로컬 MVP에 적합한 데이터 저장
- 가능하면 CSV 또는 JSON 가져오기/내보내기

첫 MVP에서 제외할 항목:

- 조직 전체용 인증/권한 시스템
- 다중 사용자 실시간 동시 편집
- 이메일 또는 메신저 알림
- 복잡한 결재 라우팅
- 외부 문서관리 시스템 연동
- 전사 배포

## 화면 톤

화면은 깔끔하고 절제되어야 한다. 불만 게시판, 공개 포럼, KPI가 과한 대시보드처럼 보이면 안 된다.

우선순위는 빠른 읽기다. 입력은 기록이 흩어지지 않을 정도로만 구조화하고, 관리자가 쓰기 부담스러울 만큼 무겁게 만들지 않는다.

UI 기준 목업은 두 가지를 함께 따른다.

- 전체 흐름: `docs/mockups/issue-board-hybrid-home-subtopic-detail-v1.png`
- 상세 페이지 구성: `docs/mockups/issue-detail-date-ledger-v5-home-button.png`

첫 화면은 A형 운영 보드 구조를 사용한다. 대분류 카드 안에 하위 주제 카드가 보이고, 각 하위 주제는 최근 날짜와 미해결 수를 간단히 보여준다.

`강종/제품 > STS` 같은 하위 주제를 클릭하면 같은 화면 아래에 끼워 넣는 방식이 아니라 별도 상세 페이지로 이동한다. 상세 페이지에는 전역 왼쪽 메뉴바를 두지 않는다. 페이지 본문은 단순하고 시인성 좋은 2영역 구성을 따른다.

- 왼쪽 영역: 해당 하위 주제의 날짜별 이력 리스트가 나온다.
- 오른쪽 영역: 왼쪽에서 선택한 날짜 이력의 상세가 열린다.

오른쪽 영역의 선택 이력 상세는 두 번째 참고 이미지의 구성과 프레임을 따른다. 상단에는 이슈 묶음 제목, 태그, 상태 버튼, 더보기 버튼을 보여주고, 그 아래에는 최초 보고일, 작성자, 관련 설비, 관련 고객, 우선순위를 가로 메타 영역으로 배치한다. 본문 영역은 세 칸으로 나눈다. 왼쪽에는 첫 번째 참고 이미지처럼 같은 이슈의 날짜별 이력 타임라인을 두고, 가운데에는 선택한 날짜의 상세 내용, 조치 사항, 남은 리스크, 첨부 파일을 크게 배치하며, 오른쪽에는 관련 이슈를 둔다.

상세 페이지 상단에는 `홈으로` 버튼과 `STS 목록으로` 버튼을 모두 둔다. `홈으로`는 대분류/하위 주제 카드가 있는 첫 화면으로 돌아가고, `STS 목록으로`는 현재 하위 주제 안의 날짜별 이력 목록 위치로 돌아가는 보조 이동이다.

가장 중요한 제품 규칙은 다음과 같다.

```text
기본 탐색은 대분류와 하위 주제 기준이다.
기본 읽기는 날짜별 이력 행 기준이다.
기본 작성은 기존 세부 이슈 아래에 날짜별 이력을 추가하는 방식이다.
```
