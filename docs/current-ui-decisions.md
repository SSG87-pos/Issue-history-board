# PosLAB 이력관리 센터 현재 UI/UX 의사결정 기록

이 문서는 다음 작업자가 가장 먼저 확인해야 하는 현재 기준 문서다. 과거 설계안이나 중간 시안보다 이 문서의 결정을 우선한다.

목표는 화면을 계속 바꾸는 과정에서 핵심 구조가 흔들리지 않게 하는 것이다. 특히 상태 표현, 날짜별 이력, 이슈별 모음, 상세 화면, 이력 추가/수정 방식은 아래 원칙을 유지한다.

## 1. 제품 목적

`PosLAB 이력관리 센터`는 연구원 내부의 기술, 운영, 제도, 투자, 시험 관련 이력을 축적하는 내부 기록 도구다.

이 앱은 공개 게시판, 민원 게시판, 무거운 결재 시스템이 아니다. 소수의 관리자 또는 관리 그룹이 이력을 일관된 형식으로 남기고, 나중에 유사 상황이 생겼을 때 과거 판단과 대응 흐름을 빠르게 찾는 것이 목적이다.

MVP 단계에서는 로컬 앱으로 운영한다. 데이터는 `localStorage`에 저장하고, JSON 내보내기/가져오기/초기화 기능을 둔다. 이후 로그인, 권한, DB, 배포를 붙일 수 있도록 데이터 구조는 확장 가능하게 유지한다.

## 2. 기본 정보 구조

현재 정보 구조는 다음 계층으로 이해한다.

```text
대분류
  하위 주제
    이슈 그룹
      세부 항목
        날짜별 이력
```

예시는 다음과 같다.

```text
강종/제품
  STS
    STS-430 표면 결함 재발
      시험 조건 편차 원인 검토
        2026-06-20 기록
        2026-06-22 기록
```

날짜별 이력은 제목 유사도로 연결하지 않는다. 같은 건인지 판단하는 기준은 같은 `issueGroupId`, `detailIssueId`에 연결되어 있는지다.

## 3. 상태 표현 원칙

상태는 두 층으로 나누어 사용한다.

사용자가 목록, 회의, 대시보드에서 보는 대표 단계는 반드시 3단계로 단순화한다.

```text
접수 / 진행 / 종료
```

세부 기록 내부에서는 더 자세한 상태를 가질 수 있다.

```text
발생 / 원인검토 / 조치중 / 검증 / 해결 / 보류
```

단, 화면에 `진행중/원인검토`, `진행/조치중`, `종료/해결`, `접수/발생`처럼 의미가 겹치는 표현을 나란히 보여주지 않는다. 목록과 대시보드는 `접수/진행/종료`만 보여준다. 세부 상태는 필요할 때 기록 내용 안에서 설명한다.

상태 점 색상은 다음을 유지한다.

```text
접수: 주황
진행: 파랑
종료: 초록
```

상태 점은 너무 작게 만들지 않는다. 사용자가 한눈에 단계 차이를 볼 수 있어야 한다. 이력 목록 제목 옆에는 색상 의미를 작은 주석 형태로 둔다.

## 4. 홈 화면 기준

홈 화면은 KPI 대시보드가 아니라 진입 화면이다. 사용자가 어느 대분류와 하위 주제를 볼지 빠르게 선택하는 것이 목적이다.

유지할 방향은 다음과 같다.

- 왼쪽 사이드바는 밝은 블루/민트 계열의 아주 약한 그라데이션을 사용한다. 너무 어둡게 되돌리지 않는다.
- 사이드바 선택 상태는 왼쪽 세로줄이 아니라 얇은 경계, 밝은 표면, 은은한 그림자로 표현한다.
- 사이드바 아이콘은 이모지가 둥둥 뜨는 느낌을 피하고, 선형 아이콘 중심으로 정렬한다.
- 본문은 대분류 카드와 하위 주제 카드 중심으로 구성한다.
- 각 하위 주제 카드는 최근 날짜와 미종료 건수 정도만 보여준다.
- 오른쪽에는 `처리 지연 이슈`와 대분류별 미해결 현황을 보조 정보로 둔다.
- 화면은 단정하지만 너무 딱딱하거나 행정 시스템처럼 보이지 않게 한다.
- 선택된 박스는 21st.dev 계열의 부드러운 ring/soft surface 느낌을 따른다. 진한 파란 채움, 굵은 테두리, `선택` 문구를 다시 넣지 않는다.

현재 대분류는 다음을 기준으로 한다.

```text
강종/제품
투자/과제
설비/시험
제도/운영
```

`고객/품질` 대분류는 MVP 시연 목적상 `제도/운영`으로 바꾼다. `제도/운영` 하위 주제 예시는 다음과 같다.

```text
운영 기준
심의/승인
데이터/보안
협업 프로세스
```

## 5. 하위 주제 상세 화면 기준

예: `강종/제품 > STS`

상세 화면은 이전 참고 이미지의 읽기 좋은 스타일을 따른다. 흰색 패널, 얇은 경계선, 차분한 여백, 작은 상태 표시를 사용한다. 글자 크기는 표와 상세 본문을 제외하면 과하게 크게 하지 않는다.

기본 구성은 다음과 같다.

```text
상단: 제목, 요약 대시보드, 주요 액션
왼쪽: 날짜별 이력 / 이슈별 모음 탭
오른쪽: 선택된 이력 상세
```

상단 액션은 `STS 목록으로`, `이력 추가`를 사용한다. `홈으로`와 `STS 목록으로`가 동시에 홈으로 가는 것처럼 보이던 문제는 피한다.

모바일에서는 날짜별 이력 또는 이슈를 누르면 상세 영역으로 자동 스크롤한다. 상세에는 `상세 닫기` 동작이 있어야 하며, 닫으면 다시 목록 영역으로 돌아올 수 있어야 한다.

## 6. 상단 이슈 대시보드

상세 화면 상단의 카드 영역은 단순 숫자 3개가 아니라 이슈 단위 현황을 보여주는 작은 대시보드로 사용한다.

현재 의도는 다음 3개 카드다.

```text
이슈 현황
미종료 관리
최근 변동
```

`이슈 현황`은 전체 이슈 수와 `접수/진행/종료` 비율을 반달형 점유율 그래프 또는 유사한 요약 시각화로 보여준다.

`미종료 관리`는 접수와 진행 상태의 이슈 수, 오래 열린 이슈를 보여준다.

`최근 변동`은 최근 기간에 갱신된 이슈와 다음 확인이 가까운 항목을 보여준다.

이 카드들은 단순 정보가 아니라 필터로 작동한다. 카드를 클릭하거나 카드 내부 단계 칩을 클릭하면 아래의 날짜별 이력과 이슈별 모음이 함께 좁혀진다.

필터 해제는 큰 전체 폭 버튼으로 두지 않는다. `날짜별 이력 / 이슈별 모음` 탭과 검색창 근처에 작은 필터 바를 두고, 활성 필터 칩과 `전체 보기` 버튼을 함께 배치한다.

`전체 보기`는 다음을 모두 초기화한다.

- 대시보드 필터
- 검색어
- 미종료 필터
- 페이지 번호

## 7. 날짜별 이력 탭

날짜별 이력은 시간이 지나면 매우 길어진다. 그래서 각 행은 간결해야 한다.

목록 행에는 다음 정보만 우선 보여준다.

- 날짜
- 이슈 그룹 또는 스티커
- `접수/진행/종료` 상태 점과 텍스트
- 해당 날짜 이력 제목 또는 요약

목록 행에서 제거할 정보는 다음과 같다.

- 긴 상세 설명
- `다음 확인 2026-06-28` 같은 후속 확인 문구
- 담당자, 상세 timestamp 등 이미 오른쪽 상세나 상단 표에서 확인 가능한 중복 정보

기본은 10개 정도까지 보여주고, 더 많으면 페이지를 넘긴다. 페이지네이션은 필요한 경우에만 보인다.

검색은 날짜별 이력과 이슈별 모음 모두에 적용된다. 검색 후 다시 전체로 돌아갈 수 있도록 `전체 보기`가 검색어까지 함께 지운다.

## 8. 이슈별 모음 탭

날짜별 이력만으로는 `STS-430 표면 결함 재발` 같은 큰 이슈 단위를 한눈에 보기 어렵다. 그래서 `날짜별 이력` 옆에 `이슈별 모음` 탭을 둔다.

이슈별 모음은 `issueGroup` 단위 목록이다.

각 행은 다음을 보여준다.

- 이슈 그룹 제목
- 스티커 또는 그룹 라벨
- `접수/진행/종료` 상태 점과 텍스트
- 이력 수
- 최근 기록일
- 담당자 또는 담당 그룹
- 현재 요약

이슈 행을 클릭하면 해당 이슈의 최신 또는 대표 상세가 오른쪽 상세 영역에 열린다.

## 9. 선택 상세 패널

상세 패널은 문서처럼 읽히는 구성이어야 한다. 사용자가 준 참고 이미지의 구조를 기준으로 한다.

상단에는 날짜와 제목을 명확히 분리한다.

```text
2026-06-20
시험 조건 편차 원인 검토
```

날짜와 제목은 줄을 나누고 글자 크기 차이를 둔다. 날짜 아래에 같은 내용을 다시 반복하지 않는다.

`박연구 · 2026-06-20T11:00:00+09:00`처럼 담당자와 날짜가 중복되는 줄은 제거한다. 담당자는 상단 메타 정보에 이미 있으면 상세 본문에서 다시 보여주지 않는다.

상세 본문 섹션은 다음 구성을 유지한다.

```text
내용
조치 사항
향후 계획
첨부 URL
```

`다음 확인일`은 상세 본문에서 삭제한 상태를 유지한다. 후속 일정이 필요하면 `향후 계획` 문장 안에 자연어로 적거나 별도 기능으로 다시 설계한다.

`내용`, `조치 사항`, `향후 계획` 같은 제목에는 아주 약한 청록/초록 계열 강조색을 사용한다. 본문은 bullet 또는 짧은 문단으로 읽기 쉽게 구성한다.

상세 우측 상단의 `접수 / 진행 / 종료` 단계는 현재 단계 텍스트가 가장 또렷해야 한다. 현재 단계는 상태별 배경과 진한 상태색을 사용한다.

```text
접수: 주황 배경/텍스트
진행: 파랑 배경/텍스트
종료: 초록 배경/텍스트
```

왼쪽 `이력 목록`의 선택 행은 상태에 맞는 연한 배경을 사용한다. 선택 점과 날짜 텍스트도 해당 상태색을 따른다. 선택된 행 안에서 왼쪽 세로선이 또렷하게 보이지 않도록 한다.

상단 메타 정보는 다음 항목을 우선한다.

```text
최초 보고일
우선순위
담당자
담당연구그룹
유관부서
```

이전의 `작성자`, `관련 설비`, `관련 고객` 중심 구성은 사용하지 않는다.

상세 우측 상단에는 상세 상태 pill 대신 `접수 / 진행 / 종료` 3단계 흐름을 보여준다. `원인검토` 같은 세부 상태 pill을 그 위치에 별도로 두지 않는다.

## 10. 첨부와 참고 링크

첨부는 파일 업로드 형태가 아니라 URL 형태로 붙이는 것을 기준으로 한다.

이력 추가/수정에서는 `첨부 URL` 입력을 제공한다. 여러 URL은 줄바꿈 또는 쉼표로 나눌 수 있다.

상세에서는 `첨부 URL` 섹션에 링크 목록으로 보여준다. 없을 때는 과한 빈 박스 대신 간단히 `첨부 URL 없음` 정도로 처리한다.

## 11. 연관 이슈 결정

`연관 이슈` 섹션은 제거한다.

초기에는 같은 대분류, 같은 하위 주제, 유사 키워드로 관련 이슈를 추천하는 방식을 검토했다. 하지만 같은 대분류로 묶으면 거의 모든 이슈가 관련 이슈처럼 보이는 문제가 있다. 현재 MVP에서는 관련 이슈를 보여주지 않는다.

다시 도입하려면 다음 중 하나가 명확해야 한다.

- 관리자가 수동으로 연결한 명시적 관계
- 동일 세부 항목 또는 동일 이슈 그룹 기반 관계
- 참고 URL 또는 문서 링크로 확인되는 관계
- 키워드뿐 아니라 설비, 공정, 강종, 원인, 조치가 충분히 일치하는 관계

그 전까지는 `연관 이슈` 메뉴, 카드, 자동 추천을 만들지 않는다.

## 12. 이력 추가와 수정

이력 추가 UI는 간결해야 한다. 복잡한 메뉴와 큰 입력 필드가 뒤섞인 느낌을 주면 안 된다.

기본 구성은 두 섹션이다.

```text
대상
기록 내용
```

`대상` 섹션에는 다음을 둔다.

- 기존 이슈 검색
- 이슈 선택
- 세부 항목 선택
- 새 세부 항목으로 기록하기

`기록 내용` 섹션에는 다음을 둔다.

- 날짜
- 세부 항목 상태
- 이 기록의 상태를 이슈 상태에도 반영
- 요약
- 상세 내용
- 향후 계획
- 첨부 URL

데이터 필드명은 기존 호환을 위해 `remainingRisk`를 계속 사용할 수 있지만, 사용자에게 보이는 라벨은 `향후 계획`을 우선한다.

이미 작성된 내용을 수정할 때는 상세 화면의 `이력 수정` 버튼을 사용한다. 같은 drawer를 수정 모드로 열고, 대상 이슈는 요약 카드로만 보여준다.

MVP에서 수정 가능한 항목은 다음이다.

- 날짜
- 상태
- 상태 반영 여부
- 요약
- 상세 내용
- 향후 계획
- 첨부 URL

MVP에서는 기존 이력을 다른 이슈나 다른 세부 항목으로 옮기는 기능은 제공하지 않는다. 재배정 기능은 데이터 정리 기능으로 별도 설계한다.

## 13. 모바일 동작

모바일에서는 좌우 2단 레이아웃을 억지로 유지하지 않는다.

기본 흐름은 다음과 같다.

```text
목록 확인
-> 날짜별 이력 또는 이슈 클릭
-> 상세 영역으로 자동 스크롤
-> 상세 닫기
-> 목록 영역으로 복귀
```

모바일에서도 상태 점, 탭, 검색, 전체 보기 버튼이 작동해야 한다. 특히 목록을 많이 읽는 화면이므로 텍스트가 잘리거나 카드가 과하게 커지지 않게 한다.

## 14. 다시 흔들리지 말아야 할 것

아래 항목은 명시 요청이 없는 한 되돌리지 않는다.

- 앱 제목은 `PosLAB 이력관리 센터`를 유지한다.
- 대표 상태는 `접수/진행/종료` 3단계로 유지한다.
- 목록에서 `진행중/원인검토`처럼 중복 의미를 같이 보여주지 않는다.
- `고객/품질` 대분류는 현재 MVP 기준에서 `제도/운영`으로 유지한다.
- `연관 이슈` 섹션은 제거 상태를 유지한다.
- 날짜별 이력 행에는 긴 본문과 다음 확인일을 넣지 않는다.
- 선택 상태를 진한 파란 박스, 왼쪽 세로줄, `선택` 문구로 되돌리지 않는다.
- 타임라인 선택 점은 파랑으로 고정하지 않고 상태색을 따른다.
- 상세 본문의 `다음 확인일` 독립 항목은 되살리지 않는다.
- 필터 해제는 큰 폭의 별도 버튼이 아니라 탭/검색 근처의 작은 `전체 보기`로 둔다.
- 상세 본문에서 날짜, 담당자, 요약을 중복 반복하지 않는다.
- 첨부는 파일 업로드가 아니라 URL 입력을 기준으로 한다.
