# 2026-06-26 이력관리 보강 정리

이 문서는 2026-06-26 후속 작업에서 확정한 요구사항과 구현 결과를 이어받기 위한 기록이다. 현재 UI/UX의 기준 문서는 `docs/current-ui-decisions.md`이며, 이 문서는 이번 변경 묶음의 맥락과 검증 결과를 보완한다.

## 1. 통합 검색

검색은 특정 필드 하나가 아니라 날짜별 이력과 이슈별 모음 전반을 한꺼번에 찾는 통합 검색으로 둔다.

검색 대상은 다음을 포함한다.

- 이력: 날짜, 요약, 상세 내용, 향후 계획, 상태, 세부 단계, 기록 유형, 작성자, 첨부 URL
- 이슈: 제목, 현재 요약, 업무 라벨, 태그, 담당자, 담당부서, 유관부서, 설비, 고객, 우선순위
- 세부 카드: 제목, 현재 요약, 태그, 담당자, 담당부서, 유관부서, 설비, 고객, 우선순위

검색 placeholder는 좁게 보이지 않도록 `검색` 또는 `이슈명, 담당자, 담당부서, 키워드`처럼 통합 검색 성격을 드러낸다.

## 2. 담당 정보

담당 정보는 이슈 전체에만 고정하지 않고 세부 카드 기준으로도 가질 수 있다. 화면에서 우선 보여줄 정보는 세부 카드 값이며, 세부 카드 값이 없을 때 이슈 값을 보조로 쓴다.

표기 기준은 다음과 같다.

- `담당연구그룹`이라는 화면 라벨은 쓰지 않고 `담당부서`로 표기한다.
- `담당자`, `담당부서`, `유관부서`는 2개 이상 입력될 수 있으므로 단일 선택 UI가 아니라 자유 입력 칸으로 둔다.
- 이력 추가와 이력 수정 drawer 모두에서 `세부 카드 담당 정보` 섹션을 제공한다.
- 기존 Excel의 `담당연구그룹` 헤더는 가져오기 호환용으로 읽되, 새로 내보낼 때는 `담당부서` 헤더를 사용한다.

## 3. 처리 지연 이슈

홈 오른쪽의 `처리 지연 이슈`는 모든 미해결 이슈가 아니라 30일 이상 열린 미해결 이슈만 보여준다.

운영 기준은 다음과 같다.

- 기준일은 이슈의 `firstOccurredAt`이다.
- 미해결 상태는 `발생`, `원인검토`, `조치중`, `검증`, `보류`다.
- 30일 이상 열린 이슈만 오래된 순으로 보여준다.
- `30일 이상 미해결` 문구는 제목 줄의 정보 아이콘 옆에 붙여 보조 설명처럼 둔다.

## 4. 대분류 이모지

새 대분류를 만들 때 홈 카드에 표시할 이모지를 선택할 수 있다.

UI 기준은 다음과 같다.

- 기본 이모지 후보를 버튼형 picker로 제공한다.
- 선택된 이모지는 버튼의 선택 상태로만 드러낸다.
- 직접 입력 칸에는 선택된 프리셋 이모지를 다시 복사해 보여주지 않는다.
- 직접 입력을 사용하는 경우에만 입력 칸에 값이 표시된다.
- 새 대분류 저장 시 선택 또는 직접 입력한 이모지를 `Category.icon`에 저장한다.

## 5. 모바일 목록

모바일에서 날짜별 이력과 이슈별 모음은 독립 카드들이 크게 흩어지는 형태보다 목록 컨테이너 안에서 둥근 행 카드가 이어지는 compact 목록에 가깝게 둔다.

이유는 긴 이력 목록을 훑을 때 큰 카드보다 compact 행 목록이 연속성을 더 잘 보여주기 때문이다. 따라서 모바일에서도 목록 컨테이너 안에서 선택 배경과 상태 메타가 분명히 보이게 하고, 행 높이가 과하게 커지지 않게 한다.

## 6. 관리자 페이지

관리자 메뉴는 사이드바에서 별도 페이지로 진입할 수 있게 한다.

현재 관리자 페이지는 다음 구성을 보여준다.

- 대분류/하위 주제 수
- 이슈/세부 카드 수
- 이력 수
- 민감 이슈 수
- JSON/Excel 데이터 내보내기와 가져오기
- 세부 카드 기준 담당 정보 표
- 접수/진행/종료 단계 현황
- 향후 관리 메뉴 자리: 분류 관리, 담당 정보 관리, 권한 관리, 보고서 양식

현재 구현에서는 분류 관리, 데이터 관리, 담당 정보 관리, 권한 관리, 옵션 관리, 보고서 양식을 관리자 페이지에서 연결한다. 대분류/하위 주제 이름과 숨김, JSON/Excel 내보내기/가져오기, 이슈/세부 항목명, 담당 정보, 대표 단계, 세부 단계/유형 표시명, 후보 순서/숨김, 업무 라벨 후보는 실제 데이터와 이력 추가/상세/보고서/Excel 흐름에 반영된다. 담당 정보는 세부 항목과 부모 이슈의 대표 담당 정보가 어긋나지 않도록 함께 저장한다. 업무 라벨은 추가/삭제/정렬을 제공한다. 세부 단계와 유형은 상태색, 단계 매핑, 보고서/Excel 호환성을 유지하기 위해 현재 단계에서는 표시명, 순서, 숨김으로 운영하고, 완전한 추가/삭제형 마스터 데이터는 기존 이력 호환성과 DB 마이그레이션을 포함해 별도 설계한다.

## 9. 2026-06-28 로그인/회원가입/보고서 정리

2026-06-28 작업에서는 POSLAB entry 로그인 화면과 보고서 화면을 현재 운영 방식에 맞게 다시 정리했다.

확정한 내용은 다음과 같다.

- 로그인 첫 화면은 `PosLAB 이력관리 센터` 제목, POSLAB 랜야드 카드, `로그인 / 회원가입` 탭만 남긴다.
- 로그인 설명 문구, 부서명/이모지 표기, `이메일 또는 아이디` 라벨은 제거한다.
- 로그인 입력 라벨은 `이메일`, `비밀번호`로 둔다.
- 회원가입은 `이름`, `이메일`, `비밀번호`만 받는다.
- 사용자가 입력한 비밀번호를 임시 비밀번호로 바꾸지 않고, 관리자 승인 시 해당 이메일/비밀번호로 활성 계정을 생성한다.
- 관리자 권한 화면의 신청 목록은 `회원가입 신청`으로 표기한다.
- 회원가입 제출 버튼은 로그인 버튼과 같은 primary 색상으로 표시한다.
- 보고서 화면의 큰 전체 미리보기 카드와 이력 목록 미리보기는 제거한다.
- 보고서 화면은 선택 조건, 작은 건수 표시, Excel/Word 다운로드 중심으로 운영한다.
- 1024px tablet landscape는 일반 노트북 L 기준이 아니라 compact 검증 폭으로 보고, 사이드바와 상세/보고서 화면을 조밀하게 유지한다.

검증은 다음으로 완료했다.

- `pnpm test`
- `pnpm build`
- `.venv/bin/python -m unittest backend.tests.test_api backend.tests.test_security`
- `pnpm test:e2e:server`
- `pnpm test:e2e:server -- --grep "server login screen|access request"`

서버 E2E 전체 실행 결과는 `25 passed, 1 skipped`였고, 로그인/회원가입 focused E2E는 `4 passed`였다. 3D 랜야드 GLTF texture blob 콘솔 메시지는 기존 렌더링 경고로 남아 있으나, 테스트에서는 앱 기능 실패로 보지 않도록 필터링한다.
